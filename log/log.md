# 局域网文件共享与同步系统阶段一实现记录

记录日期：2026-06-05

## 一、阶段目标

阶段一目标是完成最小可运行版本，实现同一局域网内设备发现和普通单文件传输。

本阶段完成内容：

- 完成配置文件。
- 实现 UDP 广播设备发现。
- 实现 TCP 文件接收服务端。
- 实现 TCP 文件发送客户端。
- 实现单文件发送。
- 实现接收端保存文件。
- 提供命令行菜单用于查看设备和发送文件。

本阶段未实现内容：

- SHA-256 哈希校验。
- 分块校验。
- 断点续传。
- 文件夹自动同步。
- SQLite 文件索引。
- TLS 加密传输。
- 权限控制和配对码。

以上功能属于后续阶段。

## 二、本次修改文件

### 1. 配置文件

新增 `config.json`：

```json
{
  "device_name": "PC-A",
  "udp_port": 9000,
  "tcp_port": 9001,
  "broadcast_ip": "255.255.255.255",
  "shared_folder": "./shared_folder",
  "chunk_size": 1048576,
  "enable_tls": false
}
```

字段说明：

- `device_name`：当前设备名称，两台电脑测试时建议设置为不同名称。
- `udp_port`：UDP 广播发现端口，默认 `9000`。
- `tcp_port`：TCP 文件传输端口，默认 `9001`。
- `broadcast_ip`：广播地址，默认 `255.255.255.255`。
- `shared_folder`：接收文件保存目录，默认 `./shared_folder`。
- `chunk_size`：文件发送时每次读取的字节数，默认 1MB。
- `enable_tls`：是否启用 TLS，本阶段固定为 `false`，暂不启用。

新增 `config.py`：

- 负责读取 `config.json`。
- 如果配置文件不存在，会自动创建默认配置。
- 自动创建 `shared_folder` 接收目录。
- 将相对路径解析为项目目录下的实际路径。

### 2. UDP 设备发现模块

新增 `discovery/udp_discovery.py`：

- 周期性发送 UDP 广播消息。
- 监听 UDP 广播端口。
- 接收其他设备发送的发现消息。
- 维护在线设备列表。
- 自动过滤本机设备，避免自己发现自己。
- 自动清理长时间未收到广播的设备。

UDP 广播消息格式：

```json
{
  "type": "DISCOVERY",
  "device_id": "设备唯一标识",
  "device_name": "PC-A",
  "ip": "192.168.1.10",
  "tcp_port": 9001,
  "status": "online"
}
```

新增 `discovery/__init__.py`，用于导出发现模块的主要类和函数。

### 3. TCP 文件传输模块

新增 `transfer/protocol.py`：

- 定义 TCP 应用层协议的元信息读写方式。
- 协议格式为：`4 字节 JSON 元信息长度 + JSON 元信息 + 原始文件字节`。
- JSON 元信息最大限制为 64KB，避免异常数据占用过多内存。

新增 `transfer/tcp_server.py`：

- 启动 TCP 服务端监听文件传输端口。
- 接收客户端连接。
- 读取文件元信息。
- 按文件大小接收原始字节并写入接收目录。
- 接收成功后向发送端返回确认消息。
- 接收失败时删除未完成文件，避免留下损坏文件。
- 保存文件时只使用 basename，防止路径穿越。
- 若接收目录已有同名文件，自动保存为 `文件名_1.ext`、`文件名_2.ext`，避免覆盖。

新增 `transfer/tcp_client.py`：

- 连接目标设备 TCP 端口。
- 发送文件元信息。
- 按 `chunk_size` 读取本地文件并发送。
- 等待接收端返回传输结果。

TCP 文件元信息格式：

```json
{
  "type": "FILE_SEND",
  "file_name": "test.txt",
  "file_size": 1024,
  "chunk_size": 1048576
}
```

接收端成功响应格式：

```json
{
  "type": "FILE_RECEIVED",
  "status": "success",
  "file_name": "test.txt",
  "bytes_received": 1024
}
```

新增 `transfer/__init__.py`，用于导出 TCP 服务端和发送函数。

### 4. 命令行入口

修改 `main.py`：

## 三、系统整体流程

### 1. 启动流程

1. 用户执行：

```bash
uv run python main.py
```

2. 程序读取 `config.json`。
3. 程序创建 `shared_folder` 接收目录。
4. 程序启动 TCP 服务端，监听 `tcp_port`。
5. 程序启动 UDP 监听线程，监听 `udp_port`。
6. 程序启动 UDP 广播线程，周期性向局域网发送设备信息。
7. 程序进入命令行菜单。

### 2. 设备发现流程

1. 每台设备启动后，每隔约 3 秒发送 UDP 广播。
2. 广播内容包含设备 ID、设备名称、IP 地址、TCP 端口和在线状态。
3. 其他设备收到广播后解析 JSON 数据。
4. 如果该设备不是本机，则加入在线设备列表。
5. 用户在菜单中选择 `1`，即可查看当前发现的在线设备。
6. 如果超过一定时间未收到某设备广播，该设备会被清理出在线列表。

### 3. 文件发送流程

1. 用户在菜单中选择 `2`。
2. 程序显示当前在线设备列表。
3. 用户输入目标设备编号。
4. 用户输入本地待发送文件路径。
5. 客户端连接目标设备的 TCP 端口。
6. 客户端发送文件元信息。
7. 客户端按 `chunk_size` 读取并发送文件内容。
8. 接收端保存文件到 `shared_folder`。
9. 接收端返回成功或失败响应。
10. 发送端在命令行显示发送结果。

### 4. 文件接收流程

1. TCP 服务端等待客户端连接。
2. 收到连接后读取 4 字节元信息长度。
3. 根据长度读取 JSON 元信息。
4. 校验请求类型是否为 `FILE_SEND`。
5. 从元信息中取得文件名和文件大小。
6. 将文件保存到 `shared_folder`。
7. 如果同名文件已存在，生成新的文件名，避免覆盖。
8. 如果连接中断或接收失败，删除未完成文件。
9. 接收成功后返回 `FILE_RECEIVED` 响应。

# 局域网文件共享与同步系统阶段二实现记录

## 一、阶段目标

阶段二将原有裸字节传输升级为可校验的分块协议，确保大文件接收完成后与发送端 SHA-256 一致。

本阶段完成内容：

- 文件元信息增加协议版本、哈希算法、文件哈希和总块数。
- 增加 `FILE_READY` 握手，避免不同协议直接混传。
- 每个分块携带序号、长度和独立 SHA-256。
- 接收端逐块确认，并在落盘后复算整文件 SHA-256。
- 接收文件先写入唯一 `.part` 文件，校验成功后原子改名。
- 发送端显示接收端已确认的传输进度。
- 增加带错误码和块编号的中文错误提示。

本阶段不包含断点续传、失败块自动重传、TLS、数据库索引和文件夹同步。

## 二、传输流程

1. 发送端流式计算文件 SHA-256。
2. 发送 `FILE_SEND`，包含 `protocol_version=2`、文件名、文件大小、文件哈希、块大小和总块数。
3. 接收端校验元信息并创建临时文件，然后返回 `FILE_READY`。
4. 发送端逐块发送 `CHUNK` 元信息和原始块数据。
5. 接收端验证块序号、长度及块哈希，写入成功后返回 `CHUNK_ACK`。
6. 所有块完成后，接收端刷新文件并重新计算磁盘文件 SHA-256。
7. 哈希一致时将临时文件原子移动到最终名称，并返回 `FILE_RECEIVED`。
8. 任一步失败时返回 `TRANSFER_ERROR` 并删除临时文件。

## 三、验收方式

1. 两台电脑同时升级并运行 `uv run python main.py`。
2. 从发送端选择目标设备和一个大于 8 MiB 的文件。
3. 确认发送端进度从第一个确认块递增到 100%。
4. 确认发送成功信息包含接收端返回的 SHA-256。
5. 分别在两端使用 `sha256sum 文件路径`，确认结果完全一致。
6. 中途关闭发送端，确认接收目录中没有残留的 `.part` 文件。

# 局域网文件共享与同步系统阶段三实现记录

记录日期：2026-06-07

## 一、阶段目标

阶段三在分块校验基础上增加持久化断点状态，使连接中断或接收端停止后可以只补传缺失块。

本阶段完成内容：

- 协议升级为 `protocol_version=3`。
- 为每个文件生成稳定的传输 ID。
- 使用 `.incoming-<transfer_id>.part` 保存分块数据。
- 使用 `.incoming-<transfer_id>.state.json` 记录文件元信息和已接收块编号。
- 接收端返回 `RESUME_REQUEST`，列出 `received_chunks` 和 `missing_chunks`。
- 发送端仍顺序读取本地文件并校验整体哈希，但网络只发送缺失块。
- 每个块的数据落盘后，再原子更新状态文件。
- 所有块完成并通过整文件 SHA-256 后，原子保存最终文件并清理状态文件。
- 服务端停止时主动关闭现有连接，使未完成传输能够进入续传流程。

## 二、断点续传流程

1. 发送端计算文件 SHA-256，发送文件名、大小、哈希、块大小和总块数。
2. 接收端根据这些元信息查找状态文件。
3. 没有状态时创建固定大小的临时文件；有状态时加载已接收块编号。
4. 接收端返回 `RESUME_REQUEST.missing_chunks`。
5. 发送端只发送缺失块。
6. 接收端按 `chunk_index * chunk_size` 定位写入，并在块数据 `fsync` 后更新状态。
7. 连接中断时保留已有有效块。
8. 重连后重复步骤 2 至 6，直到所有块完成。
9. 接收端校验整文件 SHA-256，保存最终文件并删除 `.part` 和 `.state.json`。

## 三、自动化验收

新增 `tests/test_stage_three.py`，测试过程如下：

1. 发送一个 6 块文件。
2. 接收端确认前 2 块后主动停止服务端。
3. 检查状态文件中的 `received_chunks` 为 `[0, 1]`。
4. 使用同一接收目录重启服务端。
5. 再次发送同一文件，确认只传输剩余 4 块。
6. 确认最终文件 SHA-256 一致。
7. 确认传输完成后临时文件和状态文件均已删除。

执行命令：

```bash

```

# 局域网文件共享与同步系统阶段四实现记录

记录日期：2026-06-07

## 一、阶段目标

阶段四在阶段三可靠传输基础上增加共享目录自动同步，使在线设备能够自动交换文件索引，
并同步新增和修改文件。

本阶段完成内容：

- 协议升级为 `protocol_version=4`。
- 递归扫描共享目录，跳过内部数据库、符号链接和传输临时文件。
- 使用 `shared_folder/.lan-sync/index.sqlite3` 保存 SQLite 文件索引。
- 索引记录相对路径、文件大小、修改时间、SHA-256、版本、来源设备和状态。
- 使用 `active` 与 `deleted` 状态保存当前文件和删除墓碑。
- 通过 `INDEX_BEGIN`、`INDEX_ENTRY`、`INDEX_END` 逐条交换索引。
- 按修改时间和来源设备 ID 决定同步方向。
- 使用同步模式文件传输保留相对目录，并精确覆盖较旧版本。
- 接收同步文件后立即更新本地索引，避免被扫描线程误判为本机修改。
- 自动同步所有 UDP 发现的在线设备，默认每 10 秒执行一轮。

## 二、同步规则

1. 本机有活动文件、对方没有记录时，本机发送文件。
2. 两端文件哈希相同时不传输。
3. 哈希不同时，修改时间较新的版本发送给较旧版本。
4. 修改时间相同时，来源设备 ID 字典序较大的版本胜出。
5. 删除操作只写入本机墓碑，不删除远端文件。
6. 旧于墓碑的远端文件不能恢复本机文件；晚于墓碑的新修改可以重新同步。

冲突副本、远程删除和权限控制仍属于阶段五。

## 三、协议与安全

- `FILE_SEND.transfer_mode=manual` 保留原有手动发送和同名避让行为。
- `FILE_SEND.transfer_mode=sync` 增加 `relative_path`、`modified_time_ns`、
  `version`、`source_device_id` 和 `changed_at_ns`。
- 同步接收拒绝绝对路径、`..`、内部 `.lan-sync` 路径、NUL 和符号链接逃逸。
- 传输 ID 包含同步相对路径，因此不同目录中的同名文件不会共用断点状态。
- 相同文件返回 `FILE_UP_TO_DATE`，过期版本返回 `FILE_SKIPPED`。

## 四、配置

新增配置：

```json
{
  "sync_enabled": true,
  "sync_interval_seconds": 10
}
```

程序启动顺序为：建立文件索引、启动 TCP 服务、启动 UDP 发现、启动后台同步。
退出时先停止后台同步，再关闭发现和 TCP 服务。

## 五、自动化验收

新增 `tests/test_stage_four.py`，覆盖：

- 递归索引、新增、修改、版本递增和删除墓碑。
- 内部文件、临时文件和符号链接过滤。
- 修改时间、设备 ID 和删除墓碑差异规则。
- 同步相对路径校验和路径穿越拒绝。
- 聚合大小超过 64 KiB 的逐条索引交换。
- 两个回环节点自动同步嵌套目录中的新增文件。

# 局域网文件共享与同步系统阶段五实现记录

记录日期：2026-06-07

## 一、阶段目标

阶段五在自动文件夹同步基础上增加共同同步基线、冲突副本、设备配对和权限控制。

- 协议升级为 `protocol_version=5`。
- 文件索引增加来源设备名称，并自动迁移阶段四数据库。
- SQLite 新增逐设备逐路径同步基线、设备最后同步时间和冲突日志。
- 双端都偏离共同基线时判定冲突，不依赖设备时钟判断是否并发。
- 冲突主版本按修改时间、来源设备 ID 和版本确定。
- 接收主版本前原子保留本地冲突副本，重复冲突不会生成重复文件。
- 新增持久化六位配对码、共享随机访问令牌和设备授权列表。
- 索引、手动发送和自动同步均执行权限检查。
- CLI 支持配对、查看授权、修改权限、解除授权和重新生成配对码。

## 二、权限规则

- `read`：请求文件索引。
- `write`：包含 `read`，允许手动发送和自动同步。
- `admin`：允许全部当前网络操作。
- `blocked`：拒绝全部受保护操作及重新配对。

一次正确配对会在两端保存同一访问令牌，默认权限为 `write`。重新生成配对码不会撤销
已有令牌。

## 三、冲突处理

双方交换索引时也交换各自记录的共同基线。只有一端变化时同步该端；双方变化、首次
配对即存在不同内容或基线不一致时保守判定为冲突。失败版本保存为：

```text
文件名_设备名_设备ID前8位_冲突副本_哈希前8位.扩展名
```

冲突副本随后作为普通文件同步，使两台设备最终拥有相同主文件和冲突版本。删除与修改
并发时活动文件胜出并记录冲突。

## 四、自动化验收

新增 `tests/test_stage_five.py`，覆盖配对码持久化、阶段四数据库迁移、双向配对、
未授权访问、四级权限、共同基线判断、删除与修改冲突、冲突文件名碰撞，以及两台回环
设备离线修改同一文件后的冲突副本生成、内容保全、最终收敛和幂等同步。

# 局域网文件共享与同步系统阶段六实现记录

记录日期：2026-06-07

## 一、阶段目标

阶段六增加加密传输、访问审计、异常访问检测和智能诊断 Agent。

- 协议升级为 `protocol_version=6`。
- 使用 `cryptography` 自动生成 ECDSA 自签名证书，TCP 最低使用 TLS 1.2。
- 配对时交换并保存证书 SHA-256 指纹，后续连接执行指纹绑定校验。
- UDP 发现广播增加 TLS 状态和证书指纹。
- 新增 SQLite 审计事件、轮转文本日志和异常告警。
- 新增连接诊断、冲突分析和安全审计 Agent。
- Agent 使用本地规则保证离线可用，用户手动运行时可选 DeepSeek 增强。

## 二、TLS 与身份校验

证书和私钥保存在 `shared_folder/.lan-sync/tls/`，私钥权限限制为 `0600`。配对、文件
传输、索引交换、断点续传和自动同步统一使用 TLS 连接工厂。

首次 TLS 配对通过配对码确认身份，并将握手证书指纹写入授权数据库。正常访问必须匹配
已绑定指纹；没有指纹的阶段五授权记录会提示重新配对，证书变化会以
`TLS_IDENTITY_FAILED` 拒绝。

## 三、访问审计与异常检测

审计数据保存到：

```text
shared_folder/.lan-sync/audit.sqlite3
shared_folder/.lan-sync/access.log
```

记录设备上线和下线、TLS 握手、配对、认证、索引读取、文件收发、断点续传、同步、
冲突、权限变更和异常连接。敏感字段统一替换为 `[REDACTED]`。

异常规则覆盖认证失败、畸形请求、连接突增、索引频繁读取、批量传输和被阻止设备访问。
告警只提示和记录，不自动封禁；同一规则和来源使用 10 分钟冷却。

## 四、Agent

统一结果包含摘要、风险级别、证据、可能原因和建议。

- 连接诊断检查本机 IP、UDP、TCP、同网段、目标端口和 TLS 身份。
- 冲突分析读取扩展冲突记录，比较大小、修改时间、状态、共同基线和胜出规则。
- 安全审计每 10 分钟检查 TLS、证书、共享目录、授权设备、陌生设备和近期告警。
